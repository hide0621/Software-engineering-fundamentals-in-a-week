# 第1章 ~ システム開発の全体像 ~

- システム...『**仕組み**』として体系的にまとめられたもの。コンピュータに限った話ではない。ことコンピュータにおいては、ハードウェアやソフトウェア、ネットワークなどを含めた幅広い概念のこと。

- ソフトウェア(ソフト)...コンピュータ上に保存されている電子データのこと。そのためインターネット上からダウンロードしたり、処理を実行して書き換えたりすることで簡単に入手したり変更できたりする。『**OS**』もソフトウェアの一つと考えられる。ソフトウェアはアプリにしてもOSにしても、それらのプログラムの他に設定ファイルやデータなども含んでいる。

- ハードウェア(ハード)...マウスやキーボード、ディスプレイやスピーカーだけでなく、CPUやメモリ、ハードディスク、SSDなどの内蔵されているものも含めたもの。**ソフトウェアのようには簡単には改変できない物理的な機器のこと。**

- 基幹系システム...**仕事で使うシステムのうち、それが止まってしまうと業務に多大なる影響を及ぼしてしまったり、もしくは業務が完全に止まってしまうようなシステム**のこと。『**ヒト、モノ、カネ**』と言った会社の資産を管理するようなシステム。

- 情報系システム...普段の業務を円滑に進めるために必要なシステムのこと。例えば、メールやチャット、スケジュール管理等といったものが挙げられ、それが無いと困りはするが無くても完全に業務が止まる訳でもなく、また代替手段もあるようなシステムのこと。

- 業務システム...基幹系システムと情報系システムを合わせたもの。個人で使うものではなく組織として業務で使うもの。ただし最近は『情報系システム』と同じ意味で使われる

- 基本ソフト...**OS**(**オペレーティングシステム**)のこと。これによりハードウェアの制御やコンピュータの基本的な管理を行う。またOSはハードウェアの違いを認識してくれる。電源が入ると同時に起動する。

- 応用ソフト...**アプリケーションシステム**(**アプリ**)のこと。webブラウザも応用ソフト。利用者が使いたいものを入れる形式のほか、工場出荷時に端末にOSと一緒に入れられているもの(**OSプリインストール**)、OSのインストール時に同時に入れられるもの(**OSバンドル**)もある。

- プログラム...実行ファイル(**ビルド成果物**)のこと。人間には理解しづらいがコンピュータには理解できるようになっている。利用者がプログラムを実行することでコンピュータがそれを処理し、利用者は様々な機能を使うことが出来る。

- OSS...オープンソースソフトウェアの略。**特定の企業に所属する人だけで開発するのではなくコミュニティによって開発が進められる**。開発者がライセンスを指定してソースコードを公開し、そのライセンスの範囲内で使えるようにしている。**このライセンスに従う限り無料で使用できるだけでなく、そのソースコードを変更して新たなプログラムを作成することも出来る**。また、一般的なOSSは**ソースコードを変更した際は変更したソースコードも公開することを求めるライセンス**になっている。

- 環境...OSやCPU、メモリと言ったコンピューターの5大要素(**入力、出力、記憶、演算、制御**)だったり、ネットワークなどのITインフラのこと。詰まる所、あるソフトウェアを構成するハードウェアやソフトウェア(ツールも含む)はどんなものを使っているのか、バージョンは何か、どんな構成で設計（アーキテクチャー）は何か、どんな設定なのか等と言ったもののこと。

- デスクトップアプリ...Excel、Postman、Insomnia、Kindle、Slack、Android Studio、VS Code、IntelliJ IDEA、DBeaverなどのようにパソコン本体に入れて使うアプリのこと。以前はCDやDVD経由で入れられることが多かったが、最近はインターネット経由で入れられるものが多い。そのため、ネットに繋がっていなくても使えるアプリもある。ユーザーが色々なバージョンで使う場合が想定されるため、自動アップデートが出来ない場合はサポート期間を設けて対応することが多い。

- GPU...Graphics Processing Unitの略。画像処理に特化した装置で画面に画像を表示したり、座標を変換したりするために使われる。最近はAIの研究や暗号資産のマイニングなどにも使われる。

- スマホアプリ...スマホに入れて使うアプリ。アプリストア経由で入れることが多いが、直接ビルド成果物を入れてインストールすることも出来る。様々な端末、OSに合わせて使われるようにしないといけないためテストが複雑になったり、設定もそれを踏まえてやらないといけなかったり、アプリストアもそれぞれ管理しないといけない。デスクトップアプリと同様、ユーザーは色々なバージョンで使うことが想定される。この仕様はタブレットアプリも同様だろう。

- NFC...**Near Field Communication**の略。ICチップを近付けるだけで通信が出来る技術。クレジットカード、交通系ICカード、ナナコなど。

- webアプリ...webブラウザからインターネット経由で使うアプリ。アプリ自体はwebサーバーにあり、そこにアクセスして使うことが出来る。GitHub、Sentry、Supabase、Google Cloud、Kong Konnect、Jira、コンフル、アサナ、Figma、Gather、検索エンジンなど多岐に渡る。アプリをwebサーバーに配置すればユーザーに津々浦々に最新のバージョンを提供できる良さがあるが、webブラウザの違いによって見た目や挙動が変わる可能性があったり、webサーバーを管理すると言った負担(攻撃されるリスク込み)が生まれる。

- イントラネット...企業などの組織内部だけで使うことが出来る(アクセス出来る)プライベートなネットワークのこと。インターネットで使われるものと同じプロトコル(通信規約)を使う。

- 開発の流れ...要件定義(要求分析と要件定義) → 設計 → 開発・実装(単に実装とも？) → テスト

- 要求分析...システム化にあたって発注者が持つ要望や現在課題に感じていることを整理する段階のこと。**要求定義**とも言う。

- 要件定義...要求分析によって発注者の要望を聞き出したあと、実現可能性や費用面なども考慮しつつ、システムで実現する範囲(**スコープ**)を発注者と受注者で調整して決めること。システム開発で実現する品質や範囲を要件定義の段階で決めておかないと後から要望が追加され、開発が終わらなくなってしまう。（**スコープクリーピング**）

- 要求分析と要件定義の違い...要求分析は発注者側の要望を整理するフェーズ、要件定義はそれを開発者側がどこまでやるか(開発するか)を文章に残すこと。要求分析は**why**、要件定義は**what**、基本設計・詳細設計・実装は**how**の部分。

- 設計...要件定義を経て、それをどのように作るかを考えること。『基本設計』と『詳細設計』に分けられる。
基本設計はユーザーの視点の話で、どんな画面が必要で、どんなデータが入力され、どんなデータが返ってきて、他のどんなシステムと繋がるのかを決めること。
詳細設計は開発者の視点の話で、どんな処理を実装するのか、どんなデータ構造にするのか、作ったモジュールをどのように分割するのかなどを決めること。
『入力 → 処理 →出力』のうち、基本設計は『**入力と出力**』、詳細設計は『**処理**』がスコープとなる。
一般的には基本設計では『What』、詳細設計では『How』を考えると言われている。

- 開発、実装...言語やフレームワークを選定してソースコードを書くことは勿論、サーバーを用意すること、DBやテーブル、初期データや権限周りを用意すること、ツールを選定して入れることなど、実際は開発工程におけるあらゆる領域を網羅している。

- テスト...詳細設計を満たしているかを単体テストが、基本設計を満たしているかを結合テストが、要件定義を満たしているかをシステムテストが、要求分析を満たしているかを受入テストが検証する。これを『**V字モデル**』と言ったりする。

- 運用...システムを日常的に問題なく稼働させること。具体的な作業には、バックアップの取得や攻撃が行われていないかの監視、定期的な点検でのシステムの起動や停止、再起動、ユーザーからの問い合わせ対応などが挙げられる。『**システムの安定稼働**(**障害の予防**)』が目的。

- 保守...システムの不具合の修正や機能追加のこと。具体的な作業としては、定期点検、システムのアップデートや不具合の調査、その修正や復旧作業、新しいプログラムの追加、対応状況の報告などが挙げられる。『システムの改善(最新内容の反映、障害からの復旧)』が目的。

- ウォーターフォール...要件定義(要求分析と要件定義)、設計、実装、テストと言った開発フェーズと運用フェーズが滝の流れのように一方通行になっていて、出戻りが前提になっていない開発手法のこと。
金融機関のシステムなど大規模なプロジェクトで用いられることが多く、上流の段階で仕様にミスや漏れがないがきっちりと調べて明確に定め、それをドキュメントにすることが求められる。
ただしwebアプリにおいては技術の変化も早く、ユーザーのニーズも移り変わったり多様化したりするので、この開発手法のもとでプロダクトを作ると『**完成時点が最高潮であとは衰退するだけ**』(アーキテクチャはモノリス？)と言った具合になり、プロダクトの持続的な成長には繋がらない。

- アジャイル...『繰り返し型』の開発手法で、要件定義(要求分析と要件定義)からテスト、およびリリースまでサイクルを小さい単位で繰り返すことであり、リリースの度に得られるフィードバックを受けて、また上記と同じ開発工程を経てリリースをして、フィードバックを受けてまた開発してリリースして...と言った具合の開発手法をとる。
個々の開発フェーズ(リリース含む)では最初から『車』と言った完成したプロダクトを作ることを目指すのではなく、最初は『スケボー』、次は『自転車』、次は『バイク』、最後に『車』と言った感じで『それぞれの段階で動くものの完成を目指し、徐々に完成品へと近付けていく』といった特徴がある。
不確実性を考慮して早期レビューを入れることでユーザーにより早く、より良いものを届けて、それをまたより良く育てることができる、という強みがある。
この点で言えば、**『車の部品』を作るのではなく、『車ではなくても動くもの』を作る**ことがアジャイルにおいては重要なのかもしれない。

- スクラム...ソフトウェア開発を『スプリント』という短期間で区切り、この期間で設計、実装、テストを繰り返す開発手法。スプリントの期間で作るものを明確にして、優先順位の高いものから取り組む。

- スパイラル...短期間で開発のステップを繰り返す手法であり、機能単位でそれを繰り返し、完成形に近付けていく手法。

- プロトタイプ...短期間で開発のステップを繰り返す手法であり、最初にラフな試作品を作り、そのフィードバックを受けて徐々に完成形に近付けていく手法。完成後のイメージを共有しやすいメリットがあるが、開発中に色々な追加要望がどんどん出てしまい、POCのを作るだけに終わってしまう(永遠に開発が終わらない)ことにも繋がることがある。一見この手法はアジャイルな開発手法に基づいているものと思われるが、一歩間違えるとこの手法も先のウォーターフォールのようにモノリスな感じに終わってしまうのかも。

- DevOps...開発と運用保守が同じチームで一貫して担当する体制。
開発は開発だけ、運用保守は運用保守だけを見るのではなく、プロダクトの成功・成長のために同じ方向を向いて仕事をする姿勢。
Amazon等のテック企業だと開発した人が運用まで見るのが通例だが、日本企業(JTC？)だと開発と運用がチームを組んで行うことが多いのではないか？
なおDevOpsにおいては、『お互いを尊重する』『お互いを信頼する』『失敗に対して健全な態度を取る』『非難を避ける』と言った組織文化を醸成することが重要となる。
